# Innovation Phase 1 — Step 1

## Privacy-preserving synthetic data and liquidity hotspots

Status: design output for review. This step does not implement spike detection,
operational planning, alert workflows, UI payloads, or monitoring.

## 1. Design goals

The hotspot feature must answer: “Which coarse area or agent network is under
liquidity pressure?” It must not answer: “Which real customer is involved?”

The design therefore uses:

- opaque database identifiers for joins;
- keyed pseudonyms generated outside the database;
- provider-scoped customer tokens so the same customer cannot be joined across
  providers by accident;
- coarse location cells instead of latitude, longitude, addresses, or names;
- time buckets and aggregate observations for hotspot computation;
- minimum cohort suppression before any hotspot is returned to a UI or report.

The tokens are pseudonyms, not irreversible anonymization. A production
deployment must protect the HMAC key in a secrets manager and control access to
any system that can resolve a source identifier.

## 2. Tokenization contract

The ingestion boundary receives the source identifier only long enough to create
a scoped token. It stores no source name, phone number, account number, PIN,
OTP, private key, or raw location.

```text
agent_token    = "agt_" + base32(HMAC-SHA256(agent_secret, tenant_id + ":agent:" + source_agent_id))
customer_token = "cus_" + base32(HMAC-SHA256(customer_secret, provider_id + ":customer:" + source_customer_id))
```

The provider namespace is intentionally part of `customer_token` generation.
That means two providers cannot use a shared token to correlate a customer. If
the platform later needs an approved cross-provider study, it should receive a
separate, purpose-limited research token from the identity boundary rather than
reusing provider tokens.

Each event also receives a random `event_id` generated by the ingestion service.
The event identifier is for deduplication and audit correlation; it is not a
customer identity.

## 3. Normalized SQL design

The following is the proposed extension to the current SQLite schema. It is
portable to PostgreSQL with `TEXT` replaced by `UUID` where appropriate and
with the same composite-key relationships retained.

```sql
-- No raw names, phone numbers, account numbers, latitude, or longitude appear
-- in these tables. All IDs are opaque values created at the ingestion boundary.

CREATE TABLE IF NOT EXISTS privacy_agents (
  agent_id          TEXT PRIMARY KEY,
  agent_token       TEXT NOT NULL UNIQUE,
  token_version     INTEGER NOT NULL DEFAULT 1,
  active_from       TEXT NOT NULL,
  active_to         TEXT
);

CREATE TABLE IF NOT EXISTS privacy_customers (
  provider_id       TEXT NOT NULL REFERENCES providers(id),
  customer_id       TEXT NOT NULL,
  customer_token    TEXT NOT NULL,
  token_version     INTEGER NOT NULL DEFAULT 1,
  active_from       TEXT NOT NULL,
  active_to         TEXT,
  PRIMARY KEY (provider_id, customer_id),
  UNIQUE (provider_id, customer_token)
);

CREATE TABLE IF NOT EXISTS provider_agent_scope (
  provider_id       TEXT NOT NULL REFERENCES providers(id),
  agent_id          TEXT NOT NULL REFERENCES privacy_agents(agent_id),
  scope_status      TEXT NOT NULL CHECK (scope_status IN ('active', 'closed')),
  active_from       TEXT NOT NULL,
  active_to         TEXT,
  PRIMARY KEY (provider_id, agent_id)
);

CREATE TABLE IF NOT EXISTS privacy_geo_cells (
  cell_id           TEXT PRIMARY KEY,
  grid_system       TEXT NOT NULL CHECK (grid_system IN ('h3', 'geohash')),
  resolution        INTEGER NOT NULL CHECK (resolution BETWEEN 0 AND 15),
  region_code       TEXT NOT NULL,
  release_class     TEXT NOT NULL CHECK (release_class IN ('coarse', 'restricted'))
);

CREATE TABLE IF NOT EXISTS privacy_agent_cells (
  agent_id          TEXT NOT NULL REFERENCES privacy_agents(agent_id),
  cell_id           TEXT NOT NULL REFERENCES privacy_geo_cells(cell_id),
  valid_from        TEXT NOT NULL,
  valid_to          TEXT,
  PRIMARY KEY (agent_id, cell_id, valid_from)
);

CREATE TABLE IF NOT EXISTS privacy_events (
  event_id          TEXT PRIMARY KEY,
  provider_id       TEXT NOT NULL REFERENCES providers(id),
  agent_id          TEXT NOT NULL REFERENCES privacy_agents(agent_id),
  customer_id       TEXT,
  cell_id           TEXT NOT NULL REFERENCES privacy_geo_cells(cell_id),
  event_kind        TEXT NOT NULL CHECK (event_kind IN ('cash_in', 'cash_out')),
  amount_bucket     INTEGER NOT NULL CHECK (amount_bucket >= 0),
  time_bucket       TEXT NOT NULL,
  ingested_at       TEXT NOT NULL,
  FOREIGN KEY (provider_id, customer_id)
    REFERENCES privacy_customers(provider_id, customer_id),
  FOREIGN KEY (provider_id, agent_id)
    REFERENCES provider_agent_scope(provider_id, agent_id)
);

CREATE INDEX IF NOT EXISTS idx_privacy_events_provider_time
  ON privacy_events(provider_id, time_bucket);
CREATE INDEX IF NOT EXISTS idx_privacy_events_cell_time
  ON privacy_events(cell_id, time_bucket);

CREATE TABLE IF NOT EXISTS agent_network_edges (
  provider_id       TEXT NOT NULL REFERENCES providers(id),
  source_agent_id   TEXT NOT NULL REFERENCES privacy_agents(agent_id),
  target_agent_id   TEXT NOT NULL REFERENCES privacy_agents(agent_id),
  time_bucket       TEXT NOT NULL,
  interaction_count INTEGER NOT NULL CHECK (interaction_count > 0),
  amount_bucket     INTEGER NOT NULL CHECK (amount_bucket >= 0),
  PRIMARY KEY (provider_id, source_agent_id, target_agent_id, time_bucket),
  CHECK (source_agent_id <> target_agent_id),
  FOREIGN KEY (provider_id, source_agent_id)
    REFERENCES provider_agent_scope(provider_id, agent_id),
  FOREIGN KEY (provider_id, target_agent_id)
    REFERENCES provider_agent_scope(provider_id, agent_id)
);

CREATE TABLE IF NOT EXISTS hotspot_runs (
  run_id            TEXT PRIMARY KEY,
  provider_id       TEXT REFERENCES providers(id),
  window_start      TEXT NOT NULL,
  window_end        TEXT NOT NULL,
  algorithm         TEXT NOT NULL CHECK (algorithm IN ('cell_pressure', 'network_pressure')),
  privacy_k         INTEGER NOT NULL CHECK (privacy_k >= 5),
  created_at        TEXT NOT NULL
);

CREATE TABLE IF NOT EXISTS hotspots (
  hotspot_id        TEXT PRIMARY KEY,
  run_id            TEXT NOT NULL REFERENCES hotspot_runs(run_id),
  cell_id           TEXT REFERENCES privacy_geo_cells(cell_id),
  network_key       TEXT,
  provider_id       TEXT REFERENCES providers(id),
  member_count      INTEGER NOT NULL CHECK (member_count >= 0),
  pressure_score    REAL NOT NULL CHECK (pressure_score BETWEEN 0 AND 1),
  suppressed        INTEGER NOT NULL DEFAULT 0 CHECK (suppressed IN (0, 1)),
  CHECK ((cell_id IS NOT NULL) OR (network_key IS NOT NULL))
);

CREATE TABLE IF NOT EXISTS hotspot_evidence (
  hotspot_id        TEXT NOT NULL REFERENCES hotspots(hotspot_id),
  evidence_code     TEXT NOT NULL,
  observed_value    REAL NOT NULL,
  baseline_value    REAL NOT NULL,
  contribution      REAL NOT NULL,
  PRIMARY KEY (hotspot_id, evidence_code)
);
```

### Provider boundary enforcement

`provider_agent_scope` is the authority table. The composite foreign keys on
`privacy_events` and `agent_network_edges` prevent a row from claiming that an
agent belongs to a provider when that relationship has not been explicitly
registered. Every read path must still apply a provider predicate, but the
database now rejects invalid cross-provider relationships instead of relying on
application code alone.

The proposed tables do not replace the existing synthetic demo tables yet. A
follow-up migration should either backfill these tables from the synthetic seed
or keep the two schemas separate until the new feature is approved.

## 4. Hotspot algorithm

### 4.1 Cell-pressure hotspots

Run the computation per provider and time window. The same SQL or service call
must never combine provider-scoped customer rows.

1. Bucket events into a fixed window, such as 15 minutes, and join each event to
   its coarse `cell_id`.
2. For each `(provider_id, cell_id, time_bucket)`, calculate:
   - net cash pressure: `cash_out - cash_in`;
   - event count;
   - distinct agent count;
   - e-money depletion contribution from provider balances;
   - comparison with the same cell and weekday/time-of-day baseline.
3. Suppress a candidate when `distinct agent count < privacy_k` or when the
   configured customer cohort is below `privacy_k`. Suppression returns only a
   data-quality reason, never the small cohort details.
4. Normalize the remaining features to `[0, 1]` and compute a transparent
   pressure score, for example:

   ```text
   score = 0.40 * normalized_net_cash_pressure
         + 0.25 * normalized_depletion_rate
         + 0.20 * normalized_event_count_change
         + 0.15 * normalized_baseline_deviation
   ```

5. Store the score and one row per evidence contribution in
   `hotspot_evidence`. The UI can say “cash-out pressure is 2.1× the weekday
   baseline” instead of displaying an opaque model result.
6. Return only coarse `cell_id`/`region_code`, provider scope, score, evidence,
   and cohort count. Do not return customer tokens or member-level events.

### 4.2 Network-pressure hotspots

For an approved provider scope, treat `agent_network_edges` as a weighted graph:

- node = opaque `agent_id`;
- edge weight = bounded interaction count and amount bucket;
- node pressure = local cash pressure plus provider e-money depletion;
- cluster = connected components or a community algorithm on edges above a
  minimum weight;
- network score = weighted average pressure multiplied by the fraction of
  pressured nodes.

Before writing a network hotspot, apply the same `privacy_k` rule to the number
of distinct agents. A graph cluster below the threshold is stored as
`suppressed = 1` with aggregate evidence only. Cross-provider edges are not
allowed by the provider composite foreign keys.

## 5. Output contract for the next phase

The later context-aware detector may consume a hotspot summary like this:

```json
{
  "hotspotId": "hs_demo_001",
  "providerId": "bkash",
  "regionCode": "DHK-NORTH-03",
  "window": "2026-07-12T15:00:00Z/2026-07-12T15:15:00Z",
  "pressureScore": 0.78,
  "suppressed": false,
  "evidence": [
    {"code": "cash_out_vs_baseline", "observed": 2.1, "baseline": 1.0},
    {"code": "provider_ewallet_depletion", "observed": 0.64, "baseline": 0.31}
  ]
}
```

This is an aggregate advisory input. It is not a fraud determination and does
not authorize account blocking, fund freezing, or automated transfers.

## 6. Acceptance criteria for this step

- No table stores raw customer identifiers or precise coordinates.
- Provider/customer and provider/agent relationships are explicit and
  composite-key constrained.
- Cell and network hotspots enforce a configurable minimum cohort.
- Every score has stored, human-readable evidence contributions.
- A query can produce a provider-scoped aggregate without exposing member-level
  customer data.

Approval requested: review this schema and hotspot design before Phase 1,
Step 2 (context-aware spike detection) is implemented.
